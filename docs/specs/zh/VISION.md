# 项目愿景与需求 — 主参考文档

> **来源：** 所有会话中的用户提示，整合为单一参考文档。
> **目的：** 用于检查所有规范文档是否与用户实际愿景、需求和约束一致的权威来源。
> **最后更新：** 2026-06-18

---

## 1. 我们在构建什么

**面向受监管行业中企业级 AI Agent 的确定性工作流框架。**

它不是一个预构建的产品。它是一个**参考架构 + 设计模式框架**，提供：
- 开发者实现业务逻辑的清晰接口
- 预注入的模式（状态感知提示词、确定性回退、粘滞模式、子工作流复用），负责处理"脏活累活"
- 全面的规范文档，既是技术参考，也是下游技能的面试模板

下游技能加载这些规范，并通过引导式问答帮助开发者产出完整的产品特定确定性 AI Agent 规范 — 为代码实现规划做好准备。

## 3. 要做什么

### 3.1 规范文档（当前阶段）

- [x] 高层设计（3 层架构、按节点控制、框架原则）
- [x] 意图分类（第 1 层，LLM 优先 + 关键词回退）
- [x] 状态机设计（transitions + LangGraph 融合、状态元数据、意图+状态解析）
- [x] 提取层（Extract/Validate/Transform 流水线、按接口选项）
- [x] 领域模型（实体 + 状态 + 转换模式、单一真实来源、跨工作流复用）
- [x] 路由与执行（代码执行器、决策节点、粘滞模式、子工作流、重试/errorNode、权限、工具）
- [x] 响应生成（目标设定、响应模式、目标检查、PII 清理、回环）
- [x] 工具生态（LangFlow、LangGraph CLI、LangSmith、规则引擎、MCP 服务器）
- [x] 环境配置（dev / e2e / prod）

### 3.2 规范内容要求

每个规范文档必须：
1. **仅定义接口** — 不锁定单一方案
2. **每个接口至少提供 2 个实现选项** — 附带对比矩阵
3. **将 WHAT 和 HOW 分离** — 领域模型定义 what，工作流配置定义 how
4. **包含变更日志** — 版本 + 日期 + 变更
5. **交叉引用父/子文档** — 维护文档树
6. **包含开放问题** — 不提前解决，留待架构师讨论
7. **仅包含模式 + 示例** — 不含实现代码

### 3.3 规范完成之后

- 构建下游的**规范生成器技能**，加载这些规范并与开发者面谈
- 向规范中添加面试模板 / 决策点标记
- 开始 Python 参考实现（在收到请求时）

## 4. 不要做什么

### 4.1 绝对不做

- ❌ **暂不编写实现代码。** 规范优先。仅模式 + 示例。实现延后。
- ❌ **不编写框架 API 规范。** 7 份规范已定义所有接口。AI 可从中推导出组装方式。无需单独的"如何启动"文档。
- ❌ **不提前解决。** 开放问题留待采用期间的架构师级讨论。
- ❌ **不锁定单一行业。** 框架是通用的 — 可为任何受监管行业配置。

### 4.2 应避免

- ❌ **不要重复。** 如果某个概念已在一份规范中定义，引用它；不要重新定义。
- ❌ **不要过度指定实现。** "LLM 调用 OpenAI"过于具体。"LLM 提供商（可配置）"才是正确的。
- ❌ **不要丢失技能愿景。** 每份规范都应能被下游 AI 用作面试模板。

## 6. 设计原则

### 6.1 代码规范

- 每个方法 ≤ **50 行**
- 每个文件 ≤ **1000 行**
- 执行器小而可组合，单一职责
- 复杂工作流跨文件和子工作流拆分
- 所有文档/注释使用英文；团队讨论使用中文

### 6.2 接口哲学

- 框架 = 清晰的接口 + 注入的模式
- 开发者实现业务逻辑；框架吸收脏活累活：
  - LLM 护栏（JSON 模式验证、字段存在性、类型强制）
  - 权限强制执行（按节点的工具 + 转换允许列表）
  - 重试预算（按节点重试 + 升级）
  - 审计追踪（每个决策、提取、转换都被记录）
  - 确定性回退（每个可提取字段的正则/关键词回退）
  - 状态感知（当前 FSM 状态注入每个 LLM 提示词）
  - 粘滞模式（带返回栈的阶段感知路由）
  - 子工作流复用（共享能力定义一次）

### 6.3 LLM 规则

- **所有 LLM 输出为 JSON** — 框架强制执行输出验证护栏
- **自由文本仅限于第 3 层（响应）**
- **LLM 决策需要评估** — 输入/输出测试用例，通过率 ≥95%
- **LLM 节点在节点重试预算基础上额外 +1 次重试**
- **温度 = 0** 用于提取/分类/决策；0.3 用于响应生成

### 6.4 权限规则

- **配置级** — 按节点 YAML 允许列表：`allowed_tools`、`allowed_transitions`
- **OAuth / 基于角色** — 基于认证用户范围的运行时强制执行
- **工具类别** — read、write、sensitive_data_read、dangerous_operation_write
- **`dangerous_operation_write`** 需要人工审批关卡

### 6.5 错误处理

- **所有错误 → errorNode** — 统一处理，不按类别分发
- **LLM +1 额外重试** — 耗尽后进入 errorNode
- **所有超时 → errorNode**（重试耗尽后）
- **errorNode 策略** — ask_clarify、escalate_to_human、terminate、fallback_value、retry_with_context

### 6.6 环境感知

- **3 种环境**：dev、e2e、prod
- **dev**：便宜的模型（gpt-4o-mini）、宽松的阈值、模拟 API
- **e2e**：生产模型、模拟 API、与生产相同的阈值
- **prod**：完整护栏、真实 API、硬性保护
